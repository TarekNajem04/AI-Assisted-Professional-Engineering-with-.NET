# سياسة الإصدارات

## الغاية

تحدد هذه الوثيقة آلية إنشاء الإصدارات الرسمية للمشروع.

وتهدف إلى ضمان أن يمثل كل إصدار محطة نشر مستقرة وقابلة لإعادة الإنتاج وذات قيمة حقيقية للقراء.

الإصدار ليس مجرد Git Tag.

بل حدث نشر رسمي يوثق مرحلة مهمة من تطور الكتاب.

---

# مبادئ الإصدارات

يجب أن يكون كل إصدار:

* قابلاً لإعادة الإنتاج
* قابلاً للمراجعة
* موثقاً
* مستقراً
* قابلاً للتتبع

ويمثل نقطة مرجعية يمكن العودة إليها مستقبلاً.

---

# أنواع الإصدارات

## إصدار التأسيس

مثال:

```text
v0.0.1
```

الغرض:

* نشر أساس المشروع
* نشر الوثائق التنظيمية
* نشر سياسات النشر
* نشر خارطة الطريق

---

## إصدار فصل

أمثلة:

```text
v0.1.0
v0.2.0
v0.3.0
```

الغرض:

* نشر فصل ناضج
* نشر ملفات التصدير
* توثيق محطة مهمة في تطور الكتاب

---

## إصدار صيانة

أمثلة:

```text
v0.1.1
v0.1.2
```

الغرض:

* تصحيح الأخطاء
* تحسين ملفات التصدير
* تحسين التوثيق

---

## إصدار رئيسي

مثال:

```text
v1.0.0
```

الغرض:

* نشر النسخة العامة الكاملة
* إنشاء خط أساس مستقر
* توثيق محطة رئيسية في عمر المشروع

---

# متطلبات الإصدار

قبل إنشاء أي إصدار يجب التحقق من:

* استقرار حالة المستودع.
* تحديث الوثائق ذات الصلة.
* تحديث جدول المحتويات.
* صحة الروابط الداخلية.
* نجاح عملية التصدير.

ولا ينبغي إنشاء أي إصدار من حالة غير متحققة أو غير مستقرة.

---

# قائمة التحقق قبل الإصدار

## التوثيق

التحقق من:

* README
* Roadmap
* Project Status
* Publishing Policies

---

## المخطوطة

التحقق من:

* اكتمال المحتوى المستهدف
* صحة الروابط المرجعية
* صحة روابط التنقل
* ترتيب الأقسام

---

## ملفات التصدير

التحقق من:

* PDF (EN)
* PDF (AR)
* DOCX (EN)
* DOCX (AR)

عند الحاجة للإصدار.

---

## المستودع

التحقق من:

* نظافة المستودع
* عدم وجود ملفات مؤقتة
* عدم وجود ملفات محلية غير مطلوبة
* عدم وجود مراجع مكسورة

---

# ملاحظات الإصدار

يجب أن يحتوي كل إصدار على:

* رقم الإصدار
* عنوان الإصدار
* ملخص عام
* الإضافات الرئيسية
* التغييرات المهمة
* التوجه القادم

---

# ملفات الإصدار

قد تتضمن ملفات الإصدار:

```text
PDF (EN)
PDF (AR)
DOCX (EN)
DOCX (AR)
```

ويجب أن تتوافق دائماً مع النسخة المرتبطة بالـ Tag.

---

# دورة الإصدار

```text
اكتمال المحتوى
        ↓
مراجعة تقنية
        ↓
مراجعة المستودع
        ↓
إنشاء ملفات التصدير
        ↓
إنشاء Tag
        ↓
إنشاء Release
        ↓
إعلان النشر
```

---

# ما بعد الإصدار

بعد النشر:

* تحديث Project Status
* تحديث Roadmap عند الحاجة
* نشر إعلان رسمي
* جمع الملاحظات
* تسجيل الدروس المستفادة

---

# مسؤولية الإصدار

تُنشأ الإصدارات الرسمية بواسطة القائمين على المشروع فقط.

ويمكن للمجتمع تقديم الملاحظات والاقتراحات، لكن صلاحية إنشاء الإصدارات تبقى لدى Maintainers المشروع.

---

# وثيقة حية

قد تتطور هذه السياسة مع تطور المشروع وتحسن آليات النشر والإدارة.
