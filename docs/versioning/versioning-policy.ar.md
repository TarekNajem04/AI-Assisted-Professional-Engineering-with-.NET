# سياسة الإصدارات

## الغاية

تحدد هذه الوثيقة سياسة الترقيم والإصدارات المعتمدة في المشروع.

وتهدف إلى توفير عملية إصدار واضحة وقابلة للتنبؤ طوال دورة حياة الكتاب.

يجب أن تعكس أرقام الإصدارات محطات نشر حقيقية وذات معنى، لا مجرد نشاط يومي داخل المستودع.

---

# المبدأ الأساسي

ليس كل Commit إصداراً.

وليس كل عملية نشر إصداراً.

وليس كل إصدار نسخة كاملة من الكتاب.

يميز المشروع بين المراحل التالية:

```text
Commit
    ↓
Publication
    ↓
Tag
    ↓
Release
    ↓
Edition
```

ولكل مرحلة غرض مختلف.

---

# نموذج الترقيم

يعتمد المشروع نموذجاً مبسطاً مستوحى من Semantic Versioning:

```text
MAJOR.MINOR.PATCH
```

أمثلة:

```text
v0.0.1
v0.1.0
v0.2.0
v1.0.0
```

---

# أنواع الإصدارات

## إصدارات التأسيس

تستخدم خلال مرحلة بناء المشروع.

مثال:

```text
v0.0.1
```

الغرض:

* تأسيس المستودع
* إعداد وثائق المشروع
* إعداد سياسات النشر
* إعداد سياسات المشاركة
* تجهيز البنية الأساسية للنشر

ولا تعني هذه الإصدارات اكتمال أي جزء من الكتاب.

---

## إصدارات الفصول

تستخدم عندما يصل فصل إلى مستوى مناسب من النضج.

أمثلة:

```text
v0.1.0
v0.2.0
v0.3.0
```

ويفترض أن يمثل كل إصدار تقدماً حقيقياً في محتوى الكتاب.

---

## إصدارات الصيانة

أمثلة:

```text
v0.1.1
v0.1.2
```

تستخدم من أجل:

* تصحيح الأخطاء
* إصلاح الأخطاء المطبعية
* تحسين التصدير
* تحسين التوثيق

ولا ينبغي أن تتضمن توسعات كبيرة في المحتوى.

---

## الإصدارات الرئيسية

مثال:

```text
v1.0.0
```

ويمثل:

* أول نسخة عامة كاملة
* استقرار البنية العامة
* استقرار عملية النشر
* نضج المخطوطة

---

# نشر الأقسام

القسم هو وحدة نشر.

لكنه ليس وحدة إصدار.

نشر قسم جديد:

* لا يتطلب Tag.
* لا يتطلب GitHub Release.
* لا يتطلب ملفات PDF أو DOCX.

دورة العمل النموذجية:

```text
كتابة القسم
    ↓
مراجعة القسم
    ↓
Commit
    ↓
Push
    ↓
إعلان النشر
```

---

# سياسة الـ Tags

تمثل الـ Tags محطات نشر مهمة.

ويجب إنشاؤها فقط عندما:

* يتم تحقيق محطة واضحة.
* تتوفر ملاحظات إصدار مناسبة.
* تكون حالة المستودع مستقرة.

ولا ينبغي استخدام Tags للتغييرات اليومية الاعتيادية.

---

# سياسة الـ Releases

يجب أن يحتوي كل GitHub Release على:

* عنوان الإصدار
* ملاحظات الإصدار
* ملخص العمل المنجز
* الروابط المرجعية ذات الصلة

وعند توفرها يمكن أن تتضمن:

* ملفات PDF
* ملفات DOCX

---

# ملفات الإصدارات

قد تتضمن ملفات الإصدار الرسمية:

```text
PDF (EN)
PDF (AR)
DOCX (EN)
DOCX (AR)
```

ويجب أن يتم توليد جميع الملفات من المحتوى المرتبط بالـ Tag الخاص بالإصدار.

## الإصدارات اللاحقة

تتبع الإصدارات المستقبلية النمط نفسه:

```text
v0.2.0
v0.3.0
v0.4.0
...
```

ويجب أن يمثل كل إصدار تقدماً حقيقياً وملموساً في المخطوطة.

---

# ملاحظات الإصدار

يجب أن تحتوي جميع الإصدارات على ملاحظات إصدار واضحة.

وتتضمن عادة:

* المحتوى الجديد
* التحسينات المهمة
* التغييرات الهيكلية
* محطات النشر

وتهدف هذه الملاحظات إلى تسهيل متابعة تطور المشروع مع مرور الوقت.

---

# سلامة الإصدارات

يجب أن يمثل كل إصدار حالة مستقرة وقابلة لإعادة الإنتاج.

وينبغي أن يتمكن القارئ من:

* الوصول إلى المحتوى
* إنشاء ملفات التصدير
* مراجعة تاريخ المشروع

باستخدام المعلومات المتوفرة ضمن النسخة المرتبطة بالإصدار.

---

# وثيقة حية

قد تتطور هذه السياسة مع نضج المشروع.

وأي تعديلات مستقبلية يجب أن تحافظ على التزام المشروع بإصدارات واضحة وقابلة للتنبؤ وتمثل محطات حقيقية في رحلة الكتاب.
